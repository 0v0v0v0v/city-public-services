# 城市公共便民点位整合平台技术落地计划

## 一、项目目标

本项目拟建设一个面向城市居民的公益性信息整合平台，用于集中展示城市公共便民设施信息，解决公共服务资源分散、查询效率低、使用门槛高的问题。

首版目标不是实现复杂地图能力，而是完成一个可运行、可维护、可扩展的 Web 系统，重点实现以下能力：
- 公共便民点位信息的结构化管理
- 面向用户的分类浏览、关键词搜索、详情查看
- 面向管理员的数据录入、编辑和维护
- 便民资讯发布与展示

## 二、技术建设原则

### 1. 先完成最小可用版本
首版只实现信息整合、内容展示、后台管理三类核心功能，不引入实时定位、路线规划、评论系统等复杂模块。

### 2. 采用前后端分离架构
前端负责页面渲染和交互，后端负责数据建模、业务逻辑和接口输出，便于后续扩展和多人协作开发。

### 3. 保持低部署成本
课程项目阶段优先使用 SQLite 作为数据库，减少环境配置复杂度，确保项目能在本地快速启动和演示。

### 4. 为后续扩展预留接口
虽然首版不做地图和区域筛选，但在数据结构、字段设计和接口层面保留扩展空间。

## 三、系统总体架构

系统采用前后端分离结构，整体分为三层：

### 1. 表现层
- 技术栈：Vue 3 + Vite + Tailwind CSS + Pinia + Vue Router
- 作用：
  - 展示用户端页面
  - 展示管理员后台页面
  - 管理页面状态、登录态和接口请求

### 2. 接口层
- 技术栈：FastAPI
- 作用：
  - 提供 RESTful API
  - 处理前端请求
  - 承担业务逻辑和权限校验

### 3. 数据层
- 技术栈：SQLite + SQLAlchemy
- 作用：
  - 存储分类、点位、资讯、管理员账号等数据
  - 提供 ORM 模型映射
  - 支撑基础查询、筛选、分页和写入

系统当前架构与仓库现状一致：
- 后端目录：`backend/app`
- 前端目录：`frontend/src`
- 启动方式：前端 `Vite`，后端 `Uvicorn`

## 五、核心数据模型设计

当前后端已经具备基础模型，首版可在现有结构上稳定推进。

### 1. 分类表 `categories`

建议字段：
- `id`：主键
- `name`：分类名称
- `slug`：分类标识，用于前后端识别
- `icon`：分类图标
- `cover_image`：分类封面图
- `sort_order`：排序值
- `description`：分类说明

作用：
- 统一管理点位分类
- 支持首页分类展示和筛选

### 2. 点位表 `points`

建议字段：
- `id`：主键
- `name`：点位名称
- `category_id`：所属分类
- `address`：详细地址
- `opening_hours`：开放时间
- `description`：基础简介
- `service_content`：服务内容
- `target_people`：适用人群
- `image_url`：图片地址
- `is_featured`：是否推荐
- `status`：发布状态
- `created_at`：创建时间
- `updated_at`：更新时间

说明：
- 当前字段已满足首版展示需求
- 后续若需要地图功能，可补充 `longitude`、`latitude`、`district`

### 3. 资讯表 `news`

建议字段：
- `id`：主键
- `title`：标题
- `summary`：摘要
- `content`：正文
- `published_at`：发布时间
- `status`：发布状态

作用：
- 发布平台通知、便民提醒、政策相关资讯

### 4. 管理员表 `admin_users`

建议字段：
- `id`：主键
- `username`：用户名
- `hashed_password`：加密后的密码

作用：
- 支持后台登录和管理操作鉴权

## 六、接口设计方案

首版接口按“公共接口”和“管理接口”两组划分。

## 1. 公共接口

### `GET /api/categories`
用途：获取分类列表  
返回：分类数组

### `GET /api/points`
用途：获取点位列表  
支持参数：
- `keyword`
- `category_id`
- `is_featured`
- `page`
- `page_size`

返回：
- `items`
- `total`
- `page`
- `page_size`

### `GET /api/points/{id}`
用途：获取单个点位详情

### `GET /api/news`
用途：获取资讯列表  
支持参数：
- `page`
- `page_size`

### `GET /api/news/{id}`
用途：获取资讯详情

### `GET /api/home`
用途：获取首页聚合数据  
返回内容：
- 推荐点位
- 分类列表
- 最新资讯

## 2. 管理接口

### `POST /api/admin/auth/login`
用途：管理员登录并获取 token

### `GET /api/admin/dashboard`
用途：获取后台总览数据

### 分类管理
- `GET /api/admin/categories`
- `POST /api/admin/categories`
- `PUT /api/admin/categories/{id}`
- `DELETE /api/admin/categories/{id}`

### 点位管理
- `GET /api/admin/points`
- `POST /api/admin/points`
- `PUT /api/admin/points/{id}`
- `DELETE /api/admin/points/{id}`

### 资讯管理
- `GET /api/admin/news`
- `POST /api/admin/news`
- `PUT /api/admin/news/{id}`
- `DELETE /api/admin/news/{id}`

## 七、后端实现计划

后端开发以“模型先行、接口分层、测试跟进”为原则。

- 配置项目入口和健康检查接口

当前仓库已基本完成这部分内容。

### 第二阶段：ORM 模型与数据库初始化

- 建立模型关系，如 `Category` 与 `Point` 的一对多关系
- 实现数据库初始化逻辑
- 写入演示数据和默认管理员账号

当前仓库已具备初始实现，后续重点是字段稳定性和数据完整性。

### 第三阶段：Schema 与数据校验

- 为新增、编辑接口增加字段校验
- 统一分页返回结构
- 统一错误提示信息

建议补充项：
- 对 `slug`、`status`、`page_size` 等字段增加更严格校验
- 统一管理接口和公共接口的返回结构

### 第四阶段：公共查询接口
- 实现分类查询
- 实现点位列表查询
- 实现点位详情查询
- 实现资讯列表与详情查询
- 实现首页聚合接口

关键点：
- 列表接口支持分页
- 点位接口支持关键词和分类过滤

### 第五阶段：后台管理接口

- 实现后台总览统计

关键点：
- 所有管理接口必须先通过 token 鉴权
- 删除前做好存在性判断

### 第六阶段：服务层抽离与可维护性优化
当前项目已有 `services/query.py`，后续可继续增强：
- 抽离点位筛选逻辑
- 抽离分页逻辑
- 将复杂查询从 API 层下沉到 service 层

这样可以降低路由函数复杂度，提升后续扩展性。

## 九、联调计划

联调按照“接口先通，再看页面，再查异常”的顺序进行。

### 1. 接口联调顺序
1. 登录接口
2. 分类接口
3. 点位列表接口
4. 点位详情接口
5. 资讯接口
6. 首页聚合接口
7. 后台新增与编辑接口

### 2. 联调重点
- 字段命名是否统一
- 列表分页参数是否一致
- 前端筛选参数是否能正确传入后端
- 后台创建的数据能否立即在前台展示
- 鉴权失败时页面是否正确处理

### 3. 演示闭环
联调完成后，至少要保证一条完整链路：
- 管理员登录
- 新增分类或点位
- 前台刷新后能看到新增内容
- 用户可进入详情页查看完整信息

## 十三、技术风险与应对

### 1. SQLite 扩展性有限
风险：不适合高并发或复杂事务。  
应对：课程项目阶段可接受，后续如需扩展可替换为 MySQL 或 PostgreSQL。

### 2. 接口返回结构不统一
风险：前端对接复杂，容易出现字段错误。  
应对：统一 Schema 和分页格式，联调前先确认返回字段文档。

### 3. 前后端状态不同步
风险：后台新增后前台不刷新或展示异常。  
应对：新增操作成功后主动刷新相关列表，并在联调阶段做闭环测试。

### 4. 演示数据过少
风险：虽然系统可运行，但页面观感空、答辩效果差。  
应对：每类至少准备若干条完整点位数据，并准备 3 到 5 条资讯内容。

### 5. 登录与权限逻辑不稳定
风险：后台接口调用失败，影响演示。  
应对：优先保证登录、token 存储、鉴权头注入和 401 处理稳定。
